Windows下使用的PE（Portable Executable）可执行文件

Linux系统下使用的是ELF（Executable and Linkable Format）可执行文件

两种文件都是由COFF（Common Object File Format）格式发展而来的

PE文件由DOS头、PE文件头、节表及各节数据组成：同时，如果需要引用外部的动态链接库，则有导入表：如果自己可以提供函数给其他程序来动态链接（常见于DLL文件），则有导出表

ELF文件由ELF头、各节数据、节表、字符串段、符号表组成。

节是程序中各部分的逻辑划分。.text或者.code代表代码节、.data代表数据节等。

GDB的调试方式：

（1）控制程序执行

  • r（run）：启动程序
  • c（continue）：让暂停的程序继续执行
  • si（step instruction）：汇编指令层面上的单步步入
  • ni（next instruction）：汇编层面上的单步步过
  • finish：执行到当前函数返回

（2）查看内存、表达式等

  • x/dddFFF:ddd代表长度，FFF代表格式，如”x/10gx”
  • p(print):输出一个表达式的值

（3）断点相关命令

  • b（break）：b* location，location可以为十六进制数、名称等。”*”是指中断在指定的地址，而不是对应的源代码行
  • info b:列出所有断点，每个断点会有自己的序号
  • del：删除指定序号的断点，如”del 1”
  • clear:删除指定位置的断点，如”clear *main”

(4)修改数据

  • 修改寄存器：set $rax=0x100000

  • 修改内存：set {要赋值的类型}地址=值，形如”set {int}0x405000=0x12345”

    注意：GDB不会在入口处暂停程序

Pwndbg提供了IDA的整合脚本，只需在IDA中运行Pwndbg目录下的ida_script.py,然后本机Pwndbg链接到IDA上，并使用IDA的各种功能

X64DBG支持32位和64位程序的调试，并在不断开发、添加新的功能。OllyDBG仅支持32位程序，且已经停止更新。

1.打开文件：支持用户直接将文件拖入界面，左上区域为反汇编结果的显示区域，左下区域为浏览程序内存数据的区域，右下区域为栈数据的显示区域，右上区域为寄存器的显示区域

2.控制程序运行：Ctrl+G组合键，可以跳转到目标地址；在反汇编窗口中，按F2键为切换当前地址的断点状态，按F8键为单步步过，按F7为单步步入，按F4键为运行到光标处，按F9键为运行。

常见的断点位置包括程序内的某个地址、程序调用的某个API。此外可以让程序在操作特定的某一小段内存时中断，其原理为使用CPU内建的硬件断点机制或使用Windows提供的异常处理机制的内存断点。X64DBG在内存窗口/栈窗口中选定目标地址，然后单击右键，在弹出的快捷菜单中选择”断点→硬件断点”，可以设置硬件读取和硬件写入断点；在反汇编窗口中，右击目标地址，在弹出的快捷菜单中选择”Breakpoint→Set hardware on execution”，设置硬件执行断点

3.简单的脱壳：”壳”是一种特殊的程序，对另一个程序进行变换后，利用变换的结果重新生成可执行文件。在运行时，它全部或部分还原存储在可执行文件中的变换结果，然后恢复原程序的执行。壳的存在主要是两方面的需求，压缩壳为了减小程序体积，加密壳则是为了加大破解者的逆向难度（相当于进行一个加密）。

有的壳注重对代码的压缩，从而生成更小的可执行文件，如UPX、ASPack等。有的壳注重对代码的保护，以阻碍逆向者进行分析为目的，如VMP、ASProtect等。

脱壳即为将这样的壳去除，还原为最初程序的样子。最广泛的UPX壳。UPX是一个开源的历史悠久的压缩壳。

脱壳UPX的两种办法：

  • 静态分析：UPX本身即提供脱壳器，使用命令行参数-d即可，但是有时会失败，需要切换使用正确的UPX版本。Windows下内置多个UPX版本的第三方的图形化界面UPXShell工具，可以方便地切换版本。

  • 动态方法（注意需要在Windows xp下进行）：可执行文件被操作系统载入后开始执行前，寄存器会存放一些操作系统预先填充好的值，栈的数据也会被设置，壳程序要保留这些数据，以免其被壳段代码不经意间地破坏，在转交控制权前壳需要恢复这些数据，才能让原来的程序正常执行。

    一般情况下，由于已有栈的内容是不应更改的，简单的壳会选择将这样的信息压入栈，x86的pushad可以轻松的将所有寄存器一次性压入栈，UPX也使用了这样的方式，被形象的称为保护现场。载入后发现程序最开始为pushad指令。执行后，在栈顶下读取断点，那么当程序执行完后续的还原代码操作，使用popad指令恢复寄存器时就会中断。

    故先单步执行pushad指令，再设置硬件读取断点。在OllyDBG中，右击寄存器区域，在弹出的快捷菜单中选择”HW break [ESP]”即可。x64DBG则可直接在栈窗口中利用右键快捷菜单设置。然后按”F9”运行，再次中断到一个不同的地址。然后在OD的菜单栏选择“调试→硬件断点”，然后删除。然后将光标移到最后的jmp，按F4，使得程序执行，再按F8执行跳转。此时即可dump，在OD中选择”插件→OllyDump→脱壳正在调试的进程”，在弹出的对话框中指定脱壳参数。单击”获取EIP作为OEP”按钮，再单击”脱壳”按钮，保存后完成

    除了IDA的使用，其余请在XP上完成：原因是：

    • XP后的系统中带有ASLR（地址空间随机化），程序每次启动需要重定位才可以正常执行，而恢复重定位信息难度高
    • 从Windows Vista开始，NT内核开始引入MinWin，出现了大量的api-ms-XXXXX的DLL，这导致了相当一部分依赖于NT内核特征的工具出现问题，如在Dump时使用OllyDump的导入表搜索时会受此影响
    • 从Windows 10开始，部分API有所更改，导致OllyDump的基址无法被正确填入

如何找到main函数：

  • main（）函数经常在可执行文件的靠前位置（因为很多链接器都是先处理对象文件后处理静态库）

  • VC的入口处（IDA中的start（）函数）会直接调用main（）函数，在start（）函数中被调用的函数有3个参数，并且返回值被传入exit（）函数，可以重点查看

  • GCC将main（）函数的地址传入__libc_start_main来调用main（）函数，查看调用的参数即可找到main（）函数的地址

    IDA中的一类函数与众不同：函数列表中的底色为青色，在导航条中的对应区域也会显示青色。这实际上是IDA的FLIRT函数签名识别库在起作用

    HexRays绝大多数出错的原因是与这个函数相关的某些参数设置错误

    （1）call analysis failed

    假如将sub_271010的类型从原来的”int_thiscall sub_271010(_ dword)”改为”int_cdecl sub_271010(_ dword)”,则反编译器会自动重新反编译进行刷新

    错误的根源是反编译器在寻找函数调用的参数的时候出现了错误，这时只需根据对话框前面的地址，找到出错的位置，然后修复函数的原型声明即可。

    （2）sp-analysis failed

    错误的原因是，当优化等级较高时，编译器将省略帧指针rbp的调用，转而使用rsp引用所有的局部变量。为了找到局部变量，IDA通过跟踪每条指令对rsp的修改来查找并解析局部变量。但是IDA在跟踪rsp时出现了问题，导致反编译失败

    一般情况下的根源是某个函数调用的调用约定出错，或该函数的参数个数出错，导致IDA算错了栈指针的变化量
